# 别再把 Skills 当提示词了：从安装、创建到实战，一篇讲透

如果你已经开始把 Claude、GitHub Copilot 或 VS Code Agent 用进工作流，那你迟早会遇到一个关键词：**Skills**。

很多人第一次看到它，第一反应是：

> 这不就是一个“更高级一点的提示词模板”吗？

表面上看有点像，但本质上完全不是一回事。

真正的 Skill，不是多写几句 Prompt，而是把一套任务经验、输出标准、执行步骤、参考资料，甚至脚本工具，打包成一个可以被 AI 稳定调用的**能力模块**。

说得更直白一点：

> **Prompt 决定你此刻想让 AI 做什么，Skill 决定 AI 以后如何稳定地把这类事做好。**

这也是为什么同样都在用 AI，有的人只是“偶尔省点时间”，而有的人已经开始把 AI 变成稳定的生产力系统。

这篇文章不是零散笔记，也不是命令堆砌，而是一篇完整的分享文档。我会从 4 个层面把 Skills 讲清楚：

- Skills 到底是什么，和 Prompt / Agent 有什么区别
- 在 VS Code 里怎么启用、怎么安装、怎么开始用
- 一个 Skill 到底由什么组成，为什么 `SKILL.md` 是关键
- 怎么从“使用别人的 Skill”进阶到“做自己的 Skill”

如果你一直觉得 AI 有时候很聪明，有时候又像随机发挥，那这篇文章大概率能帮你找到原因。

---

## 一、先讲结论：Skills 是 AI 工作流里最容易被低估的一层

很多人用 AI 的方式，大概都是这样：

- 遇到问题，临时发一句需求
- 回答不满意，再补几句上下文
- 结果还不够稳，再多来两轮

这种方式不是不能用，但它最大的问题是：**每次都在重新开始。**

而 Skill 的思路刚好相反：

> **把你已经验证过的做事方法，提前沉淀成标准化能力。**

比如你经常让 AI 帮你：

- 审查代码
- 设计前端页面
- 生成测试用例
- 写周报、复盘、方案文档
- 处理固定流程的数据整理任务

如果每次都要重新说明“先做什么、再做什么、最后按什么格式输出”，那其实你已经在反复消费自己的经验，却没有把这些经验沉淀下来。

Skill 的价值，就在这里：

- **让输出更稳定**：同类任务不再忽上忽下
- **让协作更省事**：不用每次从头解释
- **让经验可复用**：个人方法变成团队资产
- **让 AI 更像系统**：不只是会回答，而是会按流程办事

所以如果你问我一句：

> Skills 值不值得花时间学？

我的答案会很直接：**非常值得，而且越早越值。**

## 二、Skill、Prompt、Agent，到底是什么关系？

这三个词特别容易混。很多人一上来就一起学，最后一起糊。这里我给你一个最不容易搞混的理解方式。

### 1）Prompt：你此刻发出的指令

Prompt 是一次性的表达。

例如：

```text
帮我把这段代码优化一下。
```

它解决的是“现在要做什么”。

### 2）Skill：处理这类任务的方法

Skill 解决的不是“这一次做什么”，而是：

- 遇到这类任务时，应该怎么处理
- 重点看什么
- 输出成什么样
- 需要调用什么脚本、模板或参考资料

也就是说，Skill 处理的是**方法论层**的问题。

### 3）Agent：真正执行任务的智能体

Agent 会结合你当前的上下文、任务目标和可用工具，在适当的时候调用对应的 Skill。

所以三者的关系可以理解成：

> **Prompt 是你的需求，Skill 是你的方法，Agent 是执行方法的协作者。**

一旦这三层被串起来，AI 的表现会从“有时好用”变成“越来越可控”。

---

## 三、为什么说 Skills 比“会写 Prompt”更重要？

会写 Prompt 当然有用，但它更像是一种临场表达能力。

Skill 则更进一步，它代表的是：

> **你有没有把自己反复验证过的工作方式，整理成稳定可调用的能力。**

举个特别典型的例子。

如果你经常让 AI 帮你审查代码，每次都要重新交代这些要求：

- 先看安全问题
- 再看性能和可维护性
- 问题要分严重度
- 输出要给结论和建议
- 最后最好能生成报告

那你本质上已经在“手动扮演一个 Skill”了。

只是你还没把它真正写下来。

一旦把这些要求固化成 Skill，你以后只需要说一句：

> 帮我审查这个文件。

AI 就能更稳定地按照你定义的方式做事。

这时候，AI 带来的提升才不是“回答更顺眼”，而是**整条工作流更省力、更一致、更可复制**。

## 五、一个 Skill 的核心，其实就是 `SKILL.md`

从结构上看，一个 Skill 通常就是一个目录，而其中最关键的文件就是 `SKILL.md`。

典型目录结构如下：

```text
my-skill/
├── SKILL.md        # 必需：元数据 + 指令说明
├── scripts/        # 可选：脚本
├── references/     # 可选：参考资料
└── assets/         # 可选：模板、资源文件
```

这意味着：

- `SKILL.md` 用来定义这个 Skill 是什么、什么时候用、怎么用
- `scripts/` 可以放自动化脚本
- `references/` 可以放规范、文档、示例
- `assets/` 可以放模板和资源文件

你也可以把它理解成：

> **Prompt 是一句话，Skill 是一整个能力包。**

这就是两者的本质区别。

## 六、`SKILL.md` 到底写什么？

一个最小可用的 `SKILL.md`，通常至少要有两部分。

### 1）YAML 元数据

例如：

```yaml
---
name: pdf-processing
description: Extract text and tables from PDF files, fill forms, merge documents.
---
```

这里最关键的字段只有两个：

- `name`：Skill 的名称，尽量短、清楚、可识别
- `description`：告诉系统**什么时候该用这个 Skill**

### 2）Markdown 正文说明

正文就是 Skill 的说明书，通常要写清楚：

- 适用于哪些任务
- 触发后该怎么处理
- 推荐步骤是什么
- 最终输出需要包含哪些内容

换句话说，一个好的 `SKILL.md` 应该具备几个很重要的特征：

- **别人一看就懂**：自文档化
- **能接更多资源**：可扩展
- **方便版本管理**：可维护
- **方便共享复用**：可迁移

这也是为什么看似只是一个 Markdown 文件，实际上它却是整个 Skill 的“大脑”。

## 七、为什么代码审查 Skill 特别值得先学？

如果你要找一个最典型、最容易理解、也最适合上手模仿的 Skill 场景，那代码审查几乎是首选。

因为一个成熟的代码审查 Skill，天然就需要把很多关键内容定义清楚：

- 输入是什么
- 输出是什么
- 重点检查哪些维度
- 问题如何排序
- 报告如何呈现

最小示意可以写成：

```yaml
---
name: code-reviewer
description: 用于代码审查、质量分析、风险识别和报告生成。
---
```

正文里再继续补充：

- 审查重点：代码质量 / 安全性 / 性能 / 可维护性
- 问题分级：Critical / Medium / Minor
- 输出结果：文本报告 + HTML 报告
- 自动动作：保存文件并直接预览

这个例子真正有价值的地方，不在于字多，而在于它展示了一种很重要的思维：

> **不要只让 AI“帮我看看代码”，而要让它“按照既定标准输出结构化结果”。**

当你开始这样设计任务时，很多高频工作都会开始变得可沉淀、可复用、可放大。

## 九、怎么安装和管理 Skills？

在实际使用中，你很可能会看到不同写法，比如 `openskills` 和 `npx skills`。这并不奇怪，因为工具链在不断演进。

### 1）一种常见安装方式

```bash
npm i -g openskills
cd skill-group
openskills install anthropics/skills
```

对应界面如下：

![](skill-study-final_assets/media/image8.png)

![](skill-study-final_assets/media/image9.png)

### 2）同步技能目录

```bash
openskills sync
```

这个动作通常用于生成或同步技能目录，例如 `agents.md` 之类的索引文件。

![](skill-study-final_assets/media/image10.png)

### 3）更稳妥的理解方式

由于 Skills 相关工具迭代很快，更建议你优先确认当前环境所对应的官方命令。

如果你使用的是较新的 Skills CLI，常见命令会更接近：

```bash
npx skills find react
npx skills add <包名或仓库地址>
npx skills check
npx skills update
```

所以真正需要记住的，不是某一条命令的拼写，而是这几个核心概念：

- Skills 可以被搜索
- Skills 可以被安装
- Skills 可以被更新
- Skills 可以沉淀成你自己的能力资产

换句话说：

> **Skills 更像 AI 世界里的“能力插件”或“工作流模块”。**


![](skill-study-final_assets/media/image11.png)

- 不只是能跑
- 还要有设计感

- 还要有明确风格

这种需求如果只靠临时 Prompt，AI 往往能做出“能交差”的东西，但很难稳定地产出“有明显品质感”的结果。

而一旦有像 `frontend-design` 这样带明确审美和执行标准的 Skill，结果往往就会明显更好。

这也是 Skill 最适合发力的场景：

> **当你不仅在意“做不做得出来”，还在意“做得稳不稳、好不好、像不像你想要的风格”时。**

### 3）哪些任务最值得做成 Skill？

如果一个任务符合下面任意两条，就非常值得做成 Skill：

- 会高频重复出现
- 每次都要解释很多规则
- 你希望输出结果保持一致
- 你希望团队别人也能复用
- 你已经摸索出相对成熟的方法

比如这些场景就非常适合：

- 固定风格的代码审查
- 固定模板的周报 / 汇报 / 方案文档
- 固定流程的数据清洗与分析
- 固定风格的页面生成
- 固定规范的接口设计、测试生成、部署检查

一句话概括就是：

> **凡是你会重复做 3 次以上、并且希望结果越来越稳的高价值任务，都值得做成 Skill。**

## 十三、初学者最容易踩的 5 个坑

### 1）把 Skill 当成“更长一点的 Prompt”

这是最常见的误区。

Skill 不是 Prompt 的加长版，而是**某类任务的系统化处理方案**。

### 2）以为 Skill 越长越厉害

不是。

真正好用的 Skill，不在于篇幅，而在于：

- 触发条件是否清晰
- 执行步骤是否稳定
- 输出格式是否明确
- 示例是否足够帮助模型理解

### 3）装得越多越强

也不对。

真正高频有效的，通常都是最贴近日常工作流的那几个。

### 4）看到旧命令就直接照抄

不建议。

工具链变化很快，最好始终确认：

1. 当前官方推荐写法是什么
2. 本机环境是否支持
3. 当前插件或 CLI 版本是否匹配

### 5）只会安装别人的 Skill，不会沉淀自己的 Skill

这是最可惜的一点。

真正能长期拉开效率差距的，不是你收藏了多少 Skill，而是你有没有把自己的工作方法沉淀下来。

---

## 十四、如果你今天只做 3 件事，最推荐这 3 件

### 1）先把 Skills 功能打开

先能用，再谈后面的优化。

### 2）先装 1~3 个最贴近自己工作的 Skill

别一上来装几十个，先抓高频场景：

- 写代码多：优先代码审查、代码整理、测试生成类
- 做前端多：优先设计、美化、交互类
- 写文档多：优先总结、改写、结构化输出类

### 3）亲手做出一个自己的 Skill

这是最关键的一步。

因为从这一刻起，你不再只是“在使用 AI”，而是在**搭建属于自己的 AI 工作流资产**。

## 结语：为什么我认为 Skills 会越来越重要？

过去很多人把 AI 理解成一个“会聊天、会写字、会回答问题的工具”。

但当你真正开始使用 Skills，你会发现它的价值并不只是“更会说”，而是：

> **它越来越像一个能按你要求、按你流程、按你标准稳定做事的协作者。**

这也是为什么越来越多高效团队开始重视 Skills。

它提升的不只是表达质量，更重要的是**执行一致性**。

如果你打算真正把 AI 用进工作流、团队协作和日常生产，那么 Skills 基本是一堂绕不过去的课。

如果你现在正好刚开始接触，我最建议你下一步做的，不是继续收藏教程，而是：

> **亲手写出你的第一个 Skill。**

当你真的做出第一个可以稳定复用的 Skill，你会非常直观地感受到：AI 开始从“聊天工具”，变成“生产力系统”。
