Harness Agents：把 AI Agent 放进流水线之后，DevOps 会变成什么样？

GitHub Copilot 2026-04-30

这两个月，大家都在讲 Agent。

但很多产品说了半天，本质上还是“会聊天的助手”，最多再加一点工具调用。真正让人觉得方向有点不一样的，往往不是它能回答多少问题，而是它能不能进入真实工作流，接住那些原来只能靠脚本、人工和流程配合才能完成的活。

最近我专门看了一轮 Harness 官方资料，想弄清楚一件事：

Harness 说的 Agent，到底是一个新的 AI 聊天入口，还是一套真正能落进 DevOps 体系里的执行层？

看完之后，我的判断很明确：

Harness Agents 最值得看的地方，不是“AI 会不会写代码”，而是它试图把 Agent 直接塞进企业已有的 CI/CD 控制面里。

这件事听起来有点技术，但翻译成人话，其实很简单：

  • •不是在流水线外面再挂一个 AI 机器人
  • •不是出问题后再把日志复制给聊天窗口
  • •而是让 Agent 直接运行在 pipeline 里面
  • •用原有的权限、密钥、审批、审计和执行上下文去做事

如果这条路走通，Agent 在工程体系里的位置就会变得完全不一样。

  • •右边连接的是代码仓库、CI/CD、部署、治理、平台资源这些真实工程对象

它想做的，不是把 AI 变成一个更健谈的助手，而是把 AI 塞进工程系统的“执行路径”里。

如果你很忙，先看这 5 句话

  1. •Harness Agents 不是外挂式 AI 助手，而是 pipeline-native 的 Agent。
  2. •它最关键的创新，不在模型本身，而在执行上下文、权限继承和审计治理。
  3. •它适合的不是“陪你聊”，而是 AutoFix、Code Review、Coverage、Onboarding 这类高频工程任务。
  4. •截至 2026 年 4 月，它已经公开了清晰的产品分层和 roadmap，但整体仍带有早期产品气质。
  5. •真正值得企业关注的，不是“有没有 Agent”，而是“Agent 能不能进入现有控制面并留下治理痕迹”。

为什么这事现在值得看？

如果把时间往前拨两年，很多团队对 Agent 的感受大概是这样的：

  • •演示很好看
  • •回答也挺聪明
  • •但一旦进入企业工程环境，就总差最后一口气

这口气通常差在几个地方：

  • •拿不到真实上下文：日志、仓库、服务、环境、连接器、历史执行都散着
  • •进不了正式流程：AI 说得再对，也还是要人手工去改、去提 PR、去审批
  • •无法满足治理要求：权限、审计、可追溯、合规边界都不清楚
  • •很难变成组织资产：这次聊得好，不代表下次还能稳定复现

Harness Agents 的价值，恰恰是想把这几件事一起解决。

它不是在已有软件外面再加一个 AI 聊天框，而是试图把 Agent 放进软件交付平台本身。

这也是我觉得它值得写一篇单独分享的原因：

它讨论的已经不是“AI 会不会更聪明”，而是“AI 能不能进入受控的工程生产流程”。

先说结论：Harness Agents 想解决的，不是“更聪明”，而是“更能落地”

今天很多 AI 产品的问题，不是能力不强，而是离生产系统太远。

它们可能能：

  • •帮你解释错误日志
  • •帮你写一段修复建议
  • •帮你生成几个测试用例

但真到企业环境里，问题马上就来了：

  • •它拿不到真实上下文
  • •它没有正式权限边界
  • •它不知道审批链路
  • •它做过什么事，团队也未必看得清
  • •一旦要落到 CI/CD、部署、修复、PR 流程里，还是得绕回人工拼接

Harness Agents 的思路，明显不是再做一个“外挂式 AI 助手”，而是把 Agent 设计成 pipeline-native 的执行单元。

这意味着它从一开始就绑定了几个很关键的企业能力：

  • •执行位置：直接在 Harness pipeline 里运行
  • •上下文来源：继承 pipeline 的执行上下文、连接器、密钥和权限
  • •治理机制：走 RBAC、OPA 策略、审计日志这些企业原生能力
  • •结果形态：以 PR、评论、日志、模板等可见工件交付，而不是偷偷改东西

说白了，它想做的不是“帮工程师想一想”，而是“在企业允许的边界里，帮工程体系把一些高重复、高判断密度的工作真的跑起来”。

什么是 Harness Agents？

按照 Harness 官方文档的定义，Harness Agents 是：

运行在 Harness pipelines 内部的自治型 AI workers，用来执行软件交付生命周期里的 DevOps 任务。

这里有三个词很重要。

1）它不是聊天机器人，而是 worker

官方反复强调，Agent 不只是简单 AI chat。

在底层实现上，一个 Agent 本质上是一个 pipeline，里面包含一个或多个 AI step。它们会调用大模型，比如 Claude、OpenAI、Gemini，或者 Harness 自己的 Coding Agent，然后把这些步骤组合成一个可复用的模板。

这说明它更像是一个“可执行工作单元”，而不是一个对话框。

如果换成更容易理解的话，可以把它想成：

形态 更像什么 典型输出
普通 AI 助手 对话式顾问 建议、解释、代码片段
Harness Agent 流水线里的执行工位 PR、修复、评论、报告、模板

2）它不是外挂，而是 pipeline-native

这点几乎是 Harness 整套叙事里最核心的部分。

官方原话的重点可以概括成一句：

Agent 作为一等 pipeline step 运行，而不是外部脚本、Webhook 回调或者额外的一层机器人。

这会带来非常现实的差别：

  • •它天然拿到当前 pipeline 的上下文
  • •它能继承现有 secrets、connectors、RBAC 范围
  • •它和普通 build、test、deploy、approval step 在一个编排体系里
  • •每一步动作都在日志里可追踪、可审计

这不是一个包装差异，而是架构差异。

3）它不是黑盒，而是可 fork、可改、可版本化

官方文档里有一句我觉得特别关键：

Every agent is a pipeline.

也就是说，每个 Agent 背后其实都有明确的 YAML 定义。你可以 fork 官方 System Agent，修改它，做成自己的变体，再按模板管理、按 Git 版本化、按组织共享。

这一点很重要。

因为企业真正会买单的，通常不是“这个 Agent 很神”，而是：

  • •我能不能改它
  • •我能不能控它
  • •我能不能看清它到底怎么工作
  • •我能不能把它变成团队自己的标准能力

Harness 显然很懂这一点。

它和普通 AI 助手最大的区别，到底在哪？

如果只看宣传文案，很多 AI 产品都在说自动化、协作、智能决策。

但把 Harness Agents 拿出来单看，它和常见 AI 助手至少有四个明显差异。

一是上下文不再靠手动拼接

普通 AI 助手经常有一个老问题：

你得先把上下文喂给它，它才开始理解问题。

而 Harness Agents 的设计，是直接访问 Harness 内部的 Knowledge Graph，也就是和服务、基础设施、配置、历史执行数据相关的上下文。

官方给出的说法是，它做判断时依赖的是你的真实平台状态，而不是泛化建议。

这意味着它不是只看一段报错文本，而是有机会理解：

  • •当前服务是什么
  • •跑在哪个环境
  • •用了哪些连接器和配置
  • •之前的执行历史怎样
  • •这次失败发生在什么具体环节

对于 DevOps 场景来说，这个差别很大。

二是执行不再脱离治理

很多 Agent 演示很炫，但企业真正担心的是另一件事：

它到底能动什么？动了之后谁负责？

Harness 把治理这件事放得很前。

按照官方资料，Harness Agents 至少绑定了这些企业控制能力：

  • •Scoped permissions：继承 pipeline 的 RBAC，仅能访问允许范围内的资源
  • •OPA policy gates：有副作用的动作先过 OPA 策略
  • •Allow-listed tools：只能调用明确声明过的工具
  • •Visible artifacts only：产出以 PR、评论、日志等可见方式呈现，不做静默修改
  • •Compliance audit trail：记录模型输入、工具调用、输出和决策过程，面向合规审计

这套设计背后的潜台词很明确：

Agent 可以更主动，但不能更失控。

三是它不是只回答，而是直接进入工作流

普通 AI 助手更像“建议层”。

Harness Agents 更像“执行层”。

官方文档里反复出现的一种工作模式是：

  1. •pipeline 触发 agent
  2. •agent 分析仓库、PR 或失败执行
  3. •agent 生成修改或报告
  4. •agent 发起 PR 或写入评论
  5. •开发者审核并合并

这个闭环很朴素，但也很重要。

因为它没有跳过人，只是在“人工全程亲手做”与“完全自动放权”之间，找了一个更现实的位置。

四是它强调可观察，而不是神秘智能

官方文档里有一句挺直给：

No hidden prompts. Full reasoning chains in pipeline logs.

它想表达的是：

  • •你能看到它看到了什么
  • •你能看到它调用了什么
  • •你能看到它产出了什么
  • •你能回放它在 pipeline 里的行为链路

对企业来说，这可能比“模型多强”还重要。

用一张表看懂：Harness Agents 和普通自动化/AI 助手有什么差别？

维度 传统脚本自动化 普通 AI 助手 Harness Agents
执行位置 外部脚本或额外服务 对话窗口 Pipeline 原生步骤

权限边界 自己维护 通常较弱 继承 RBAC、资源范围和审批体系

这张表其实能说明一件事：

Harness 想切的，不是“AI coding assistant”那条赛道，而是“企业级受控 Agent automation”这层。

官方已经公开了哪些典型 Agent？

截至 2026 年 4 月底，Harness 官方公开资料里，已经给出了比较明确的一批 System Agents 和典型用例。

我把最值得关注的几类整理成更容易理解的版本。

1）AutoFix：流水线挂了，它先去修

这是一个很直观的场景。

当 CI/CD 失败时，AutoFix Agent 会分析失败信息、最近代码变更和执行上下文，然后生成修复方案，验证修复结果，再通过 pull request 的方式把修改交给开发者审核。

官方给它的价值点很直接：

  • •缩短故障恢复时间
  • •减少开发者手动排查 CI 问题的时间
  • •保持 pipeline 更稳定
  • •缩短反馈闭环

如果你带过研发团队，会知道这类问题往往非常消耗注意力：不是难，而是碎，而且高频。这个场景很适合 Agent 先打一版底稿。

2）Code Coverage：不是提醒你补测试，而是直接补

这类 Agent 的价值很容易理解。

它会去分析仓库、识别测试覆盖率缺口，然后自动生成单元测试，目标是把整体覆盖率拉到更高水平，再自动发起 PR 并附带覆盖率报告。

官方资料里甚至写到了示例阈值：

  • •总体覆盖率达到 90%+
  • •单文件覆盖率达到 80%+

它不是只告诉你“这里测试不够”，而是尝试把测试补上。这一点非常像 Agent 场景该有的样子：少一点提醒，多一点交付。

3）Code Review：从 lint 往前走一步

官方对 Code Review Agent 的描述是，它不只是做规则检查，而是结合 PR diff、问题上下文、测试结果和团队规范，给出更有判断性的审查意见。

它会关注：

  • •变更摘要
  • •风险点
  • •风格问题和反模式
  • •可重构建议
  • •缺失测试或文档

这里最值得关注的不是“AI 能不能替代 code review”，而是：

它开始尝试把代码审查从机械检查，往带上下文的工程判断推进。

当然，这个边界还需要继续观察。但方向上，它明显不是在和 lint 工具竞争。

4）Manifest Remediator / Helm Chart Autofix：把部署失败修到更靠前

这两个场景更偏 CD 和平台工程。

官方文档里提到的能力包括：

  • •分析 Kubernetes deployment 失败事件
  • •识别 manifest 里的 API 版本、selector、mount、resource limit 等问题
  • •修复 Helm chart 的 values、template、版本不匹配等错误
  • •给出修正后的 YAML、patch 或 PR

这类问题本来就适合“有上下文、能读配置、能生成改动”的 Agent 去做初步处置。

5）Onboarding：新仓库接进来，不再全靠平台团队手搓

这个场景我个人觉得非常有意思。

官方说 Onboarding Agent 可以扫描仓库、推断构建和部署拓扑，然后自动创建 CI/CD pipeline，再交由人工审批后激活。

如果这块做得稳，它能明显缓解平台团队很常见的一类工作：

  • •每来一个新项目，就要重新搭一套流水线
  • •团队经验不一致，模板质量参差不齐
  • •交付速度取决于平台同学有没有空

Agent 在这里的价值，不是完全替代平台工程，而是把“重复但还要讲究一点判断”的部分标准化。

6）升级、治理、安全修复类 Agent

官方资料还提到了几类比较典型的高价值场景：

  • •Library Upgrade：扫描依赖并发起安全升级或版本迁移 PR
  • •React Upgrade / 多生态升级：处理框架迁移、依赖升级、兼容性调整
  • •Feature Flag Cleanup：扫描过期 feature flag，删除无效代码路径
  • •Vulnerability / Zero Day Remediation：根据漏洞扫描结果自动生成修复 PR

这些活的共同特点都很像：

  • •足够重复
  • •足够耗时
  • •规则不少
  • •但又不完全是死脚本能搞定的

这恰好就是 Agent 最适合切入的一层。

这些 Agent 真正改变的，不只是效率，而是工作切分方式

很多人第一次看这类产品，会下意识问一句：

它到底能替代多少人力？

我觉得这个问题不算错，但也不算最关键。

更准确的问法也许是：

它会把团队里哪些工作，从“人肉串流程”改成“人定边界、Agent 执行、人工复核”？

如果按这个角度看，Harness Agents 影响最大的，通常是这几类活：

1）频繁发生，但又不值得每次高级工程师亲自下场的活

比如：

  • •覆盖率补齐
  • •常见 CI 故障修复
  • •依赖升级
  • •过期 feature flag 清理

这类事情单次看都不大，但对团队注意力消耗非常重。

2）涉及判断，但判断模式相对稳定的活

比如：

  • •Kubernetes manifest 配置纠错
  • •PR 代码质量分析
  • •安全扫描结果对应修复
  • •初始 pipeline 生成

这类事情并不完全是死规则，但也不需要每次从零思考。

3）以前只能靠“经验丰富的人”快速处理的活

这类场景一旦沉淀成模板，团队的能力分布会发生变化。

原来是：

  • •碰到问题，找最懂的人

未来更可能变成：

  • •先让 Agent 跑一版
  • •再由懂的人做最终判断

这听起来只是流程优化，但本质上是在改团队协作的颗粒度。

Harness 不只是一个 Agent，它其实在搭一整套 AI 工程体系

如果只看 Harness Agents 这个页面，你会以为它只是 Harness AI 里的一个新模块。

但从官方文档目录看，Harness 的想法显然更大一些。

截至目前，Harness AI 相关公开能力已经不止一个点，而是一组相互衔接的能力层：

  • •Harness Agents：运行在 pipeline 里的自治型 Agent
  • •Harness AI Code Agent：跑在 IDE 里的编码助手，支持代码生成、聊天、上下文引用、API Spec 生成等
  • •DevOps Agent：偏向流水线生产力和平台交互
  • •Release Agent：偏向发布、目标控制、feature operations
  • •Support Agent：偏向产品和文档问答
  • •Harness MCP Server：把 Harness 平台能力通过 MCP 暴露给外部 AI agent
  • •Skills / Prompting / Chat Memory / Code Search：围绕使用体验的辅助层

换句话说，Harness 并不是只做“一个能修 CI 的 Agent”。

它更像是在搭一个分层结构：

  • •IDE 内有 Code Agent
  • •平台内有 pipeline-native Agents
  • •平台外可以通过 MCP 把能力接给别的 Agent 系统
  • •中间再用 Knowledge Graph、权限、审计、策略把它们串起来

从产品思路上说，这比单点 AI 功能要完整得多。

从架构角度看，Harness 在搭什么？

官方文档里，对 Harness Agents 的结构描述可以压缩成三层：

1）运行时层：Harness Pipelines

这是执行底座。

Agent 直接运行在 Harness pipeline engine 里，因此天然继承：

  • •job 语义
  • •并行能力
  • •失败策略
  • •回滚能力
  • •审批节点
  • •连接器和密钥体系

这一层的意义是：不用额外维护一套新的 Agent runtime。

2）智能层：LLM + Knowledge Graph

这是它“为什么不像普通机器人”的关键。

如果 Agent 只能看到 prompt，它只能给出通用建议。

如果 Agent 能看到：

  • •pipeline 历史
  • •服务和环境关系
  • •仓库和配置
  • •平台对象与资源

那它做的就不再只是泛化问答，而是更贴近真实工程判断。

3）工具层：MCP + Harness APIs

这一层决定了 Agent 到底能“做什么”。

Harness 文档里把外部能力调用、平台资源访问和 MCP Gateway 都纳入了工具层语义。也就是说，Agent 不是只有语言理解能力，它还在往“受控调用系统能力”的方向走。

如果只用一句话概括这套架构，我会这么说：

Harness 不是把大模型接进 DevOps，而是把大模型包进一个可执行、可治理、可追踪的工程容器里。

这张架构图，最值得盯住哪几个点？

结合官方文档和你的这张图，我觉得有 4 个点最值得在分享里点出来。

1）Agent 不再悬浮在系统外面

很多 AI 产品最大的问题是“聪明，但飘”。

它理解不错，建议也不错，但不在系统里，所以接不住真正的执行链路。

Harness 的解法就是把 Agent 变成流程内部节点。

2）MCP 在这里不是噱头，而是扩展边界

如果一个 Agent 只能访问自家封闭能力，扩展性会很快见顶。

MCP 的价值在于，它让 Agent 和外部资源、工具、平台能力之间，有了更标准化的交互方式。

当然，Harness 的重点不只是“能接”，而是“接了以后还能控”。

3）治理和权限不是补丁，而是主设计

从 RBAC、OPA、allow-listed tools 到 compliance trail，Harness 把治理放得很前。

这意味着它不是先做个 AI demo，再补企业能力；而是从一开始就想按企业软件的要求来设计 Agent。

4）模板化是长期价值的来源

短期看，AutoFix 很酷，Code Review 很酷。

长期看，更重要的是：

  • •它们是不是能被 fork
  • •能不能被组织复用
  • •能不能逐步沉淀成团队标准模板

一旦能做到这一点，Agent 的价值就不只是“功能”，而是“资产”。

为什么这件事值得企业研发团队认真看一眼？

如果你站在企业研发管理、平台工程或者 DevOps 视角看，Harness Agents 值得关注的点大概有三层。

第一层：它在试图重写“脚本自动化”的边界

过去很多工程自动化，本质上都靠脚本。

脚本当然很有价值，但脚本也有明显边界：

  • •规则清楚时很好用
  • •情况一复杂就开始堆 if/else
  • •一旦涉及判断、权衡、上下文推理，就会快速脆弱

Harness Agents 试图补上的，就是脚本和人工之间那一层灰度地带。

也就是那些：

  • •不能只靠固定规则
  • •但也不值得每次都拉人处理
  • •既要理解上下文，又要遵守工程治理

的工作。

这类工作，在 DevOps 体系里其实非常多。

第二层：它在把 Agent 放进“有控制力的地方”

很多团队现在不是不想上 Agent，而是不敢把 Agent 真接进生产流程。

原因不复杂：

  • •权限怎么控
  • •行为怎么查
  • •风险怎么拦
  • •输出怎么审
  • •责任怎么界定

Harness 的一个优势在于，它没有试图绕开这些问题，而是直接拿自家平台的 RBAC、OPA、审计、模板、连接器体系去兜底。

这未必意味着它已经完美，但至少说明它理解企业采用 AI 时最难的部分，不在 demo，而在治理。

第三层：它在往“可复用的工程能力模板”方向走

官方把 Agent 设计成模板、步骤组、可 fork 的 pipeline 定义，这件事很关键。

因为企业最后需要的通常不是一百个孤立 Agent，而是：

  • •一套可版本化的团队能力库
  • •一套可跨项目复用的工程标准件
  • •一套可持续治理的 AI 自动化资产

如果 Agent 只能当场跑一次，它的价值更多是工具。

如果 Agent 能沉淀成模板库，它才开始接近平台能力。

如果你准备内部分享，可以怎么讲这件事？

如果你是要把这篇文章发到公众号，或者转给团队做内部讨论，我觉得最适合的讲法不是“介绍一个新产品”，而是从下面这三个问题切入。

问题一：为什么脚本自动化不够了？

因为很多工程问题不只是执行，还包含判断：

  • •看日志不是难，难在结合上下文找原因
  • •补测试不是难，难在知道该补哪里
  • •修配置不是难，难在知道改动是否安全

问题二：为什么普通 AI 助手也不够？

因为它通常不在正式执行路径里：

  • •看得到建议
  • •但接不住审批
  • •写得出答案
  • •但接不住治理

问题三：为什么 Harness Agents 值得重点观察？

因为它给出的答案不是“再来一个助手”，而是：

把 Agent 放进已有软件交付平台的控制面里，用原有平台能力去约束和放大它。

这三个问题一旦讲清楚，读者就不容易把它当成“又一个 AI 插件”。

也别神化：Harness Agents 现在更像一个很有方向感的早期产品

看完整体资料之后，我觉得也要保持一点克制。

截至 2026 年 4 月底，Harness Agents 很值得关注，但它更像一个方向非常明确、架构表达很完整、正在快速补齐能力的早期产品。

有几个点我觉得应该客观看。

1）它目前仍处在 Limited Preview 信号很强的阶段

官方产品页上明确写了 Limited Preview。这意味着：

  • •能力在快速演进
  • •实际可用范围和稳定性还要看客户环境
  • •一些能力已经清楚，但大规模成熟度还需要时间验证

2）官方 roadmap 很诱人，但也写明了是 forecast

根据当前官方文档：

  • •2026 年 2 月：Foundation Release
  • •2026 年 4 月：Agent extensibility（MVP 1）
  • •2026 年下半年：事件驱动触发、多 Agent 编排、增强版 MCP Gateway、Marketplace 等

这些方向都很对，但是否按节奏全部落地，还是要继续观察。

3）它真正的护城河，不在“有没有 Agent”，而在“上下文 + 治理 + 平台整合”

Agent 本身，越来越不是稀缺品。

真正难的是：

  • •你有没有真实上下文
  • •你有没有执行控制面
  • •你有没有企业治理能力
  • •你有没有可复用模板体系

我个人觉得，Harness 如果后面能站住，靠的不会是“我们也有 Agent”，而是“我们把 Agent 放进了软件交付平台的骨架里”。

什么时候适合试，什么时候先别急？

我觉得可以很现实地分成两类。

适合开始评估的团队

  • •已经有比较成熟的 CI/CD 体系
  • •平台工程或 DevOps 团队本来就在维护大量流水线
  • •希望降低重复运维和研发效能成本
  • •对权限、审计、变更追溯要求比较高
  • •愿意把 Agent 当“受控执行能力”而不是“玩具”

可以先观望的团队

  • •流水线本身还不稳定，基础工程标准化没完成
  • •团队规模很小，手工处理成本还不高
  • •目前更需要的是通用 coding assistant，而不是平台级 Agent
  • •对 Harness 平台生态没有接入计划

说得直白一点：

如果你的问题主要是“个人写代码快不快”，它未必是优先级最高的答案；如果你的问题是“组织级软件交付怎么更稳、更省、更可控”，它就值得认真看。

一个更现实的判断：谁会最先对 Harness Agents 感兴趣？

我觉得最有可能率先认真评估它的，不会是单兵开发者，而是这些角色：

1）平台工程团队

他们最容易被打动的，不是 AI 写代码，而是：

  • •新项目接入能不能更快
  • •公共流水线模板能不能更稳
  • •低价值重复工作能不能少一点
  • •治理成本会不会被放大

2）DevOps / SRE / Release 团队

他们会更关心：

  • •CI 失败能不能自动止血
  • •配置错误能不能提前修正
  • •发布流程里能不能多一层智能辅助
  • •出问题时是否有完整审计链路

3）研发效能负责人

他们更可能从这些维度看：

  • •MTTR 是否下降
  • •覆盖率是否提升
  • •代码审查效率是否改善
  • •平台团队服务吞吐量是否更高
  • •团队是否能把 AI 自动化沉淀成资产

如果只是个人用 AI 写写代码，Harness Agents 未必是第一优先级。

但如果你管的是一个团队、一套平台、几十上百条流水线，那这个方向就值得持续跟。

一个适合公众号结尾的判断

AI 进入工程体系，过去更多是“辅助编码”这条线。

Harness Agents 让我看到的，是另一条更偏平台和组织能力的路线：

  • •不只是帮工程师写得更快
  • •而是帮软件交付系统运转得更顺
  • •不只是给建议
  • •而是把建议变成受控执行

这两条路并不冲突，但后者显然更难，也更有企业价值。

最后一句：Harness Agents 让我看到的，不是“又一个 AI 功能”，而是 Agent 开始长进工程系统里了

过去一段时间，很多 Agent 产品更像是在已有软件外面包了一层智能。

Harness 这次更像是在做另一件事：

不是让 AI 在系统外面给建议，而是让 AI 在系统里面按规则做事。

这两者的差别，短期看可能只是产品定位不同； 但长期看，背后决定的其实是 Agent 到底能不能真的进入企业生产流程。

所以如果你问我，Harness Agents 现在最值得关注的点是什么。

我会给一个很朴素的回答：

它把 Agent 放进了 pipeline、权限、审计和模板体系里。

这一步不一定最炫，但很可能更接近真正能落地的那条路。

而且一旦这条路跑通，后面变化的可能就不只是“自动修一点 CI”，而是整个软件交付体系里那部分重复、碎片化、但又需要一点判断力的工作，会被重新切分一遍。

这件事，值得继续看。
